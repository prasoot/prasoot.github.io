PATTERN 01

Pure Proxy

App A invokes App B's synchronous API through APIM, which acts as a transparent pass-through layer — forwarding the request unchanged and returning the response to the caller without any transformation or orchestration.

// SEQUENCE
01
App A sends HTTP Request

Consumer calls the APIM endpoint with payload and auth headers (e.g. API Key / OAuth token).

02
APIM Applies Gateway Policies

Authentication, rate limiting, logging and quota checks execute — no payload transformation.

03
APIM Forwards to App B

Request is proxied as-is to the backend. APIM may inject backend credentials transparently.

04
App B Returns Response

Backend processes the call and sends back the HTTP response synchronously.

05
APIM Relays Response

Response is passed back to App A unchanged. APIM logs the completed transaction and status code.

06
App A Receives Result

Consumer processes the synchronous response. Total round-trip stays within SLA timeout window.

// When to Use

  • Backend API contract is already standardised — no mapping needed
  • Latency is critical; transformation overhead is unacceptable
  • Centralised security enforcement without touching App B
  • Simple API exposure to external consumers or partners
  • Rapid on-boarding of existing internal services to a managed gateway

// Constraints & Considerations

  • No payload mapping — consumer must understand App B's schema
  • APIM becomes a single point of failure; HA config is mandatory
  • Synchronous coupling — App A blocks until App B responds
  • Backend changes directly impact consumers (no contract buffer)
  • Not suitable for long-running or fire-and-forget workloads
